# ATLAS AI 프로젝트 인수인계 문서

> 이 파일은 ChatGPT 웹에서 논의한 ATLAS AI의 사업 방향과 개발 상태를 Codex 앱으로 전달하기 위한 기준 문서다.
> Codex는 작업을 시작하기 전에 이 문서를 먼저 읽고, 실제 로컬 파일과 Git 상태를 확인한 뒤 작업한다.

## 1\. 프로젝트 기본 정보

* 회사명: **ATLAS**
* 메인 브랜드: **ATLAS AI**
* 핵심 철학: **AI를 직접 만드는 회사에만 머무르지 않고, 사람들이 AI를 활용해 원하는 결과를 얻도록 돕는 AI 활용 전문 기업**
* 핵심 메시지: **Find the right AI. Build the right system.**
* 현재 단계: 무료 AI 비교·추천 플랫폼 MVP 구축
* 초기 목표: 무료 정보와 진단 기능으로 이용자, 검색 트래픽, 신뢰 및 실제 사용 데이터를 확보
* 장기 목표: 전문 AI 관리·컨설팅, 보안형 AI, 기업 시스템 통합 솔루션 회사로 성장

## 2\. 초기 시장 진입 방향

초기 핵심 고객은 인플루언서, 1인 창작자, 프리랜서 및 소상공인으로 설정한다.

우선 제공할 가치:

1. 자동 마케팅에 적합한 AI 추천
2. 숏폼·홍보 영상 제작 AI 추천
3. 콘텐츠 기획, 이미지, 영상, 문구 제작 도구 비교
4. 목적·예산·경험 수준을 반영한 간단한 진단
5. 추천 도구를 실제로 사용하는 순서와 실행 가이드 제공

초기에는 수익화를 서두르기보다 이용자 반응과 추천 성과를 검증한다. 이후 개인 상담, 기업 컨설팅, 구축 및 구독 서비스로 확장한다.

## 3\. 성장 단계별 수익 구조

1. **무료 플랫폼**: AI 비교, 추천, 활용 가이드, 뉴스·콘텐츠, 무료 진단으로 고객 확보
2. **개인 AI 컨설팅**: 1:1 상담, 맞춤 AI 추천, 프롬프트 설계, 업무 자동화 상담, 프리미엄 진단
3. **기업 AI 컨설팅**: AI 도입 분석, 업무 프로세스 분석, 활용 전략, PoC 및 프로젝트 컨설팅
4. **Enterprise Solutions**: 스마트팩토리, 물류, 헬스케어, 교육, 리테일, 금융, 공공·스마트시티 구축 및 유지보수
5. **SaaS**: ATLAS AI Dashboard, Workflow, Agent 관리, 모니터링을 월·연 구독으로 제공
6. **Academy**: 개인·기업·정부·공공기관 교육, 온라인 강의, 교재 및 자격 과정
7. **Marketplace**: 기업과 AI 전문가 연결, 중개·프로젝트·파트너 수수료
8. **파트너십**: AI·클라우드·하드웨어·교육기관 제휴, 리셀링 및 공동사업
9. **광고**: 플랫폼 성장 후 배너, 스폰서 및 제휴 광고
10. **API**: ATLAS 추천·비교 기능을 기업용 API로 제공

첫 유료 상품 후보는 **AI 활용 설계 상담**이다. 고객의 목표, 예산, 경험 수준을 진단하고 적합한 도구 조합과 2주 실행 계획을 제공한다.

## 4\. 자체 AI 모델과 자체 시스템의 구분

초기에는 GPT·Claude 같은 거대언어모델을 직접 학습할 필요가 없다.

* 자체 거대언어모델: 초기에는 불필요
* ATLAS 자체 시스템: 반드시 필요
* ATLAS 자체 데이터: 서비스 운영과 함께 축적
* ATLAS 추천 엔진: 데이터가 쌓이면서 고도화
* 보안형 소형 모델·미세조정 모델: 기업 수요와 사업성이 확인된 뒤 검토

GPT, Claude, Gemini 등은 작업 엔진이고, ATLAS AI는 다음 과정을 지휘하는 시스템이다.

1. 사용자 목적과 조건 질문
2. 민감정보·개인정보 입력 방지 또는 제거
3. 적합한 외부 AI 선택
4. 동일한 조건으로 결과 요청
5. 품질·비용·속도·난이도 비교
6. 사용자 상황에 맞는 최종 활용 순서 추천
7. 추천 결과, 선택, 만족도 및 실제 성과 저장

초기 추천 점수 기준:

* 가격 적합도: 30%
* 사용 편의성: 25%
* 결과물 품질: 25%
* 자동화 수준: 20%

웹사이트에서 \*\*“ATLAS AI 자체 모델이 분석합니다”\*\*라고 표현하지 않는다. 다음처럼 정직하게 설명한다.

> ATLAS AI는 다양한 AI 모델과 서비스의 결과를 자체 평가 기준으로 비교하고, 사용자의 목적에 맞는 활용 방법을 추천합니다.

## 5\. 기술 방향

### 현재 프런트엔드

* Next.js 16.2.10
* React
* TypeScript
* Tailwind CSS
* npm
* Node.js 24.18.0
* Git 2.55.0

### 향후 백엔드·데이터베이스

* FastAPI 백엔드
* PostgreSQL 데이터베이스
* OpenAI, Anthropic 및 필요 시 Gemini API 연동
* API 키는 브라우저 코드에 저장하지 않고 백엔드 환경변수로 관리
* 프런트엔드·백엔드·데이터베이스 환경을 분리
* 개발·테스트·운영 환경변수를 분리하고 `.env.example`만 Git에 포함
* 민감정보, 실제 API 키 및 고객 데이터는 Git에 커밋하지 않음

백엔드와 DB를 추가할 때는 Python 가상환경을 만들고 프런트엔드 Node 환경과 분리한다.

## 7\. Windows 실행 방법

PowerShell 실행 정책으로 `npm.ps1`이 차단될 수 있으므로 Windows에서는 다음 명령을 우선 사용한다.

```powershell
npm.cmd run dev -- --hostname 127.0.0.1
```

브라우저 접속 주소:

```text
http://127.0.0.1:3000
```

의존성을 다시 설치해야 할 때:

```powershell
npm.cmd install
```

운영 빌드 검사:

```powershell
npm.cmd run build
```

`.next`와 `node\\\_modules`는 생성 파일이므로 다른 저장장치로 복사하거나 GitHub에 올릴 필요가 없다.

## 8\. 현재 홈페이지 구현 상태

현재 확인된 주요 경로:

* `/`: 메인 홈페이지
* `/ai`: AI 검색 및 카테고리 필터
* `/compare`: 최대 3개 AI 선택 비교
* `/diagnosis`: 4단계 무료 AI 진단과 추천 결과

공통 AI 데이터:

```text
src/data/ai-tools.ts
```

AI 이름, 설명, 난이도, 요금 안내, 추천 대상, 장점 및 주의사항은 가능한 한 이 파일에서 한 번에 관리한다.

이미 작동을 확인한 기능:

* AI 이름과 목적 검색
* 카테고리 필터
* 비교할 AI 선택·해제
* 최대 3개 비교 제한
* 진단 질문 선택, 이전·다음 이동 및 결과 표시
* `/`, `/ai`, `/compare`, `/diagnosis` 페이지 접속
* 과거 시점의 `npm.cmd run build` 성공

## 9\. 현재 해결해야 할 홈페이지 작업

ChatGPT 웹 대화에서 마지막으로 확인한 문제와 다음 작업:

1. 여러 페이지에 중복된 헤더와 푸터를 공통 구성요소로 분리

   * `src/components/site-header.tsx`
   * `src/components/site-footer.tsx`
2. 내부 이동은 일반 `<a>`보다 Next.js `Link`를 사용
3. 모바일 화면에서도 메뉴를 열 수 있도록 모바일 메뉴 추가
4. 메인 페이지의 “무료 진단 시작하기” 링크를 `#contact`가 아니라 `/diagnosis`로 수정
5. 목적별 카테고리 카드는 `/ai`의 해당 카테고리 결과와 연결
6. 활용 가이드 카드의 임시 `#contact` 링크를 실제 가이드 페이지 또는 명확한 준비 중 상태로 교체
7. 장식용 절대 위치 요소에는 필요 시 `pointer-events-none`을 적용해 클릭 방해 가능성을 제거
8. 고정 헤더가 앵커 제목을 가리지 않도록 `scroll-margin-top` 적용
9. 수정 후 모든 페이지와 버튼을 직접 확인하고 운영 빌드 검사

주의: ChatGPT 작업공간에서 만든 임시 수정 파일은 Windows 로컬 프로젝트에 자동 반영되지 않는다. Codex는 실제 로컬 `atlas-ai-web` 파일을 먼저 검사하고 수정해야 한다.

## 10\. 권장 코드 구조

```text
src/
├─ app/
│  ├─ page.tsx
│  ├─ ai/page.tsx
│  ├─ compare/page.tsx
│  ├─ diagnosis/page.tsx
│  └─ guides/page.tsx
├─ components/
│  ├─ site-header.tsx
│  ├─ site-footer.tsx
│  └─ ui/
└─ data/
   └─ ai-tools.ts
```

원칙:

* 헤더·푸터·버튼 등 반복 UI는 부모 공통 구성요소로 관리
* 각 페이지는 해당 페이지에만 필요한 내용과 상태를 담당
* AI 정보는 공통 데이터 파일을 단일 기준으로 사용
* 페이지 파일 하나가 지나치게 커지면 의미 단위의 자식 구성요소로 분리
* 기존 사용자 변경을 확인하지 않고 덮어쓰지 않음

## 11\. 데이터·보안·공개 정책

* 모델과 도구에 중립적인 추천 구조 유지
* 추천 이유와 평가 기준을 설명할 수 있어야 함
* 가격, 기능 및 이용 조건은 공개 전 각 서비스 공식 페이지에서 확인
* 고객 입력이 외부 AI 사업자 서버로 전송될 수 있음을 고지
* 개인정보처리방침, 이용약관, 국외이전 고지, 삭제 방법 및 차단 정책 준비
* API 키 또는 원본 API 접속 권한을 재판매하지 않음
* 추천 결과를 맹신하게 만들지 않고 한계와 사실 확인 필요성을 함께 안내

## 12\. 백업 규칙

* 홈페이지 코드는 GitHub에 커밋·푸시
* 중요한 변경 후에는 사용자에게 GitHub 백업을 안내
* 전체 사업 자료는 향후 외장하드에도 별도 백업
* OneDrive 무료 용량이 부족하므로 개발 프로젝트의 주 저장소로 사용하지 않음
* 삭제 전에 로컬 실행, `git status`, GitHub 원격 저장소 및 빌드 결과를 확인

권장 백업 명령:

```powershell
git status
git add .
git commit -m "작업 내용을 설명하는 메시지"
git push
git status
```

마지막 `git status`에서 다음과 같은 상태를 확인한다.

```text
nothing to commit, working tree clean
```

## 13\. Codex 작업 규칙

Codex는 다음 순서로 작업한다.

1. 이 문서와 실제 프로젝트 파일을 읽는다.
2. `git status`로 기존 변경사항을 확인한다.
3. 수정 대상과 예상 결과를 초보자도 이해하기 쉬운 한국어로 짧게 설명한다.
4. 관련 없는 파일과 사용자 변경사항은 건드리지 않는다.
5. 한 번에 확인 가능한 범위로 수정한다.
6. TypeScript 오류, 린트 및 운영 빌드를 확인한다.
7. `/`, `/ai`, `/compare`, `/diagnosis`와 새 경로를 점검한다.
8. 사용자 확인 후 커밋·푸시를 안내한다.
9. 중요한 작업이 끝나면 반드시 백업을 상기시킨다.

삭제, 강제 초기화, 대규모 덮어쓰기처럼 복구가 어려운 작업은 사용자 확인 없이 실행하지 않는다.
